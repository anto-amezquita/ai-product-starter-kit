# Stage 2 — Product direction

## Goal

Turn the idea into a product with a clear user, problem, value, and first boundaries.

## Explore

- primary user
- sharpest problem
- job to be done
- value proposition
- what success looks like
- what makes the idea meaningfully different
- what belongs in version one
- what should be out of scope
- product principles
- anti-goals

## Useful questions

- "Who feels this problem most sharply?"
- "What are they trying to do when this problem appears?"
- "What would be meaningfully better for them if this existed?"
- "What is the smallest version that would still feel genuinely useful?"
- "What should we deliberately not build in the first version?"
- "What should this product never become?"

## When helpful, offer contrasts

### User focus
- power users vs beginners
- individuals vs teams
- creators vs consumers
- internal tool vs public product

### Product shape
- simple utility
- guided workflow
- collaborative platform
- content product
- automation layer

### First-version strategy
- narrow but excellent
- broader but shallow
- manual behind the scenes
- automated from the start

## Output from this stage

Draft content for `docs/product-north-star.md`:

- product summary
- problem
- audience
- user jobs
- value proposition
- product principles
- desired future
- anti-goals
- initial scope
- open questions

## Completion checklist

Before moving to Stage 3, confirm you have all of these:

- Product can be described in one clear paragraph
- Primary user identified
- Core problem named
- Value proposition stated
- First version scope defined
- At least one anti-goal named (what the product should not become)

## Stage checkpoint

When the completion checklist is satisfied:

1. Show the progress indicator with Stages 1–2 marked ✓ and Stage 3 marked →.
2. Write or update `decisions/kickoff-checkpoint.md` with the current stage, decisions, assumptions, and open questions.
3. Ask: "Ready to continue to Stage 3 — Brand character, or would you like to pause here?"

If the user pauses, let them know the checkpoint has been saved to `decisions/kickoff-checkpoint.md` — starting a new `/project-kickoff` session will pick up where they left off automatically.
